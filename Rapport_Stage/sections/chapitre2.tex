\chapter{Contexte, Problématique et Solution Proposée}

\section*{Introduction}
Ce chapitre présente le contexte général dans lequel s'inscrit ce projet de fin d'études, les enjeux de la qualité des données dans le secteur bancaire, ainsi que la problématique spécifique rencontrée au sein de CIH Bank. Nous décrirons ensuite l'architecture du pipeline de données existant avant de présenter la solution retenue et les outils mobilisés.

\textbf{Note de périmètre :} les contrôles de Data Quality développés dans ce projet s'appliquent exclusivement à la \textbf{zone RAW} du Data Lake. Cette zone est la copie exacte et non transformée des données issues du système opérant Oracle NOVA. Contrôler la qualité à ce niveau revient à mesurer directement la qualité des données à la source, avant toute transformation.

\section{Contexte général : la donnée au cœur de la banque digitale}

\subsection{La transformation digitale du secteur bancaire}

Le secteur bancaire connaît depuis plusieurs années une transformation profonde, portée par la digitalisation des services, l'essor du big data et l'émergence de l'intelligence artificielle. Dans ce contexte, les institutions financières s'appuient de plus en plus sur leurs systèmes d'information pour analyser les comportements clients, optimiser leurs processus internes et piloter leurs activités de manière efficace.

La donnée est ainsi devenue un actif stratégique de premier plan. Pour CIH Bank, les données clients constituent le socle de nombreuses activités critiques : l'octroi de crédit, la segmentation commerciale, la conformité réglementaire (KYC, FATCA, LAB-FT) et le pilotage de la performance. La fiabilité de ces décisions dépend directement de la qualité des données sur lesquelles elles reposent.

\subsection{La qualité des données : un enjeu stratégique et réglementaire}

La qualité des données (\textit{Data Quality}) est la mesure dans laquelle les données sont aptes à satisfaire les besoins pour lesquels elles sont utilisées. Elle s'évalue selon plusieurs dimensions reconnues par le standard DAMA (Data Management Association) [REF] :

\begin{itemize}
    \item \textbf{Complétude} : les champs obligatoires sont-ils renseignés ?
    \item \textbf{Validité} : les valeurs respectent-elles les formats et référentiels autorisés ?
    \item \textbf{Cohérence} : les informations inter-colonnes sont-elles logiquement compatibles ?
    \item \textbf{Fraîcheur} : les données sont-elles à jour après chaque cycle de chargement ?
    \item \textbf{Unicité} : existe-t-il des doublons sur les identifiants ou les enregistrements ?
\end{itemize}

Dans le domaine bancaire, une mauvaise qualité des données peut avoir des conséquences graves :
\begin{itemize}
    \item \textbf{Risques réglementaires} : des données KYC incomplètes ou incohérentes exposent la banque à des sanctions de Bank Al-Maghrib (BAM) et des organismes de conformité internationaux.
    \item \textbf{Risques métiers} : des données socio-professionnelles (CSP) erronées faussent les modèles de scoring crédit et la segmentation commerciale.
    \item \textbf{Risques opérationnels} : des anomalies non détectées se propagent dans l'ensemble du système d'information, contaminating les rapports de reporting et les tableaux de bord décisionnels.
\end{itemize}

\section{Le domaine Tiers : périmètre métier du projet}

\subsection{Définition du Référentiel Tiers}

Le \textbf{Référentiel Tiers} constitue le référentiel central de CIH Bank regroupant l'ensemble des informations relatives aux personnes physiques (PP) et morales (PM) en relation avec la banque. Ce référentiel est la source de vérité pour toutes les applications métiers : les crédits, la comptabilité, le reporting réglementaire et les outils de CRM.

Notre projet se concentre sur le périmètre des \textbf{Personnes Physiques (PP) Clients}, qui regroupe l'ensemble des clients particuliers et MRE (Marocains Résidant à l'Étranger) actifs de CIH Bank, soit plus de 1,15 million de clients. Le filtre appliqué en production est : \texttt{customer\_type = 1} et \texttt{code\_marche IN (1, 8)}.

\subsection{Étape 1 — Système opérant : Oracle NOVA}

La source de données est le système de \textit{Core Banking} \textbf{Oracle NOVA}, qui héberge le schéma \texttt{REFERENTIELTIERS}. Les tables sources principales utilisées sont :

\begin{itemize}
    \item \texttt{CCM\_PARTY} : informations générales du tiers (identifiant, numéro client, marché, état)
    \item \texttt{CCM\_NATURALPERSON} : lien entre le tiers et sa description de personne physique
    \item \texttt{CCM\_NPDESCRIPTION} : données d'état civil et KYC (nom, prénom, date de naissance, nationalité, FATCA)
    \item \texttt{CCM\_JOB} : données socio-professionnelles CSP (secteur, profession, CSP, employeur)
    \item \texttt{CCM\_PORTFOLIO} : portefeuille commercial (gestionnaire associé)
    \item \texttt{CCM\_CONTEXT} : contexte de création du dossier client (agence, utilisateur)
\end{itemize}

\subsection{Étape 2 — Ingestion RAW vers HDFS/Hive}

Le transfert quotidien des données depuis Oracle NOVA vers le Data Lake est orchestré par \textbf{Apache Airflow}. Le DAG principal du domaine Tiers est \textbf{\texttt{dailyow22}}, qui s'exécute chaque soir à \textbf{22h00}. Il utilise \textbf{Apache Sqoop} en mode \textit{OverWrite} (OW) Full pour importer l'ensemble des tables \texttt{CCM\_*} vers la base Hive \texttt{novatiers}, dans le répertoire HDFS \texttt{/data/raw/novatiers/}, au format ORC.

Un second DAG, \textbf{\texttt{nova\_referentieltiers\_clients}}, alimente en parallèle la base Hive \texttt{raw\_nova\_referentieltiers}, qui est la base \textbf{directement consommée par Dremio} pour la construction des vues analytiques.

\subsection{Étape 3 — Transformation STAGING}

Un job \textbf{Apache Spark} (DAG \texttt{referentieltiers\_ingestion}, script \texttt{staging\_novareferentieltiersclients.sh}) transforme et charge les données depuis la zone RAW vers la zone STAGING, dans la base Hive \texttt{staging\_nova\_referentieltiers}, au format Parquet/ORC.

\subsection{Étape 4 — Virtualisation Dremio}

\textbf{Dremio} expose les données de staging via \textbf{10 vues virtuelles} dans le schéma \texttt{staging-nova-referentieltiers} :

\begin{itemize}
    \item \texttt{v.Customers} (source : \texttt{CCM\_PARTY})
    \item \texttt{v.natural\_person\_description} (source : \texttt{CCM\_NPDESCRIPTION})
    \item \texttt{v.customer\_job} (source : \texttt{CCM\_JOB})
    \item \texttt{v.portfolio} (source : \texttt{CCM\_PORTFOLIO})
    \item \texttt{v.customer\_creation\_context} (source : \texttt{CCM\_CONTEXT})
    \item \texttt{v.customer\_segmentation} et \texttt{v.customer\_segmentation\_members}
    \item \texttt{v.operational\_posts} (source : schéma \texttt{staging-e\_params})
\end{itemize}

\subsection{Étape 5 — Vue finale CIHONE (28 colonnes)}

La vue analytique finale \textbf{CIHONE} résulte d'une jointure entre les 10 vues Dremio, filtrée sur \texttt{customer\_type=1} et \texttt{code\_marche IN (1,8)}, pour produire 28 colonnes décrivant chaque client PP actif de CIH Bank. Cette vue alimente les outils de reporting (Tableau) et les analyses métiers.

\section{Problématique}

\subsection{Absence de contrôle qualité automatisé sur le pipeline}

Bien que ce pipeline de données soit robuste et opérationnel, il ne dispose d'aucun mécanisme automatisé de contrôle de la qualité des données. Les données transitent quotidiennement depuis Oracle NOVA vers Dremio \textbf{sans aucune validation intermédiaire} sur les colonnes critiques KYC et CSP.

Cette absence de contrôle génère plusieurs problèmes identifiés dans le domaine Tiers :

\begin{itemize}
    \item \textbf{Incohérences genre / civilité} : des clients de genre masculin associés à une civilité féminine (MME, MLLE) ou inversement, signalant une saisie erronée dans NOVA.
    \item \textbf{Données CSP massivement nulles ou parasites} : les colonnes \texttt{professional\_sector}, \texttt{occupation} et \texttt{socioprofessional\_category} présentent des taux de valeurs nulles significatifs. Le champ \texttt{employer} contient de nombreuses valeurs parasites non normalisées telles que « AUTRE », « LUI MEME » ou « BIEN VU ».
    \item \textbf{Dates de naissance approximatives} : une proportion notable de clients enregistre une date de naissance au 1er janvier, signalant une saisie par défaut faute d'information précise.
    \item \textbf{Non-conformité FATCA} : des clients de nationalité américaine ne disposent d'aucun identifiant FATCA, exposant la banque à un risque réglementaire critique.
    \item \textbf{Risque de fraîcheur} : aucune alerte ne se déclenche si le batch \texttt{dailyow22} échoue ou se termine en retard, laissant Dremio exposer des données potentiellement obsolètes.
\end{itemize}

\subsection{Absence d'historisation et de traçabilité}

Les anomalies détectées manuellement ne sont ni tracées ni historisées. Il est donc impossible de mesurer l'évolution de la qualité des données dans le temps, d'identifier des tendances dégradantes ou de prouver une amélioration après une action corrective menée par les Data Stewards dans NOVA.

\subsection{Absence d'outillage pour les Data Stewards}

Les \textbf{Data Stewards} et \textbf{Data Owners} du domaine Tiers n'ont pas accès à un outil centralisé leur permettant de visualiser les anomalies, de les prioriser par criticité et de suivre l'avancement des corrections. La remédiation est aujourd'hui réactive et non structurée.

\section{Solution proposée}

\subsection{Vue d'ensemble de la solution}

Pour répondre à ces problématiques, nous proposons de concevoir et de mettre en œuvre un \textbf{pipeline automatisé de Data Quality (DQ)} déclenché quotidiennement après chaque exécution du DAG \texttt{dailyow22}, basé sur la bibliothèque Python \textbf{Soda Core}. La solution s'articule autour de quatre composantes majeures :

\begin{enumerate}
    \item \textbf{Moteur de contrôle DQ (Soda Core)} : un script Python exécutant des règles de qualité paramétrables sur les tables Hive des zones RAW et STAGING du domaine Tiers.
    \item \textbf{Couche de persistance (HDFS/Hive)} : des tables Hive dédiées pour historiser quotidiennement les scores DQ et le détail des anomalies détectées.
    \item \textbf{Exposition analytique (Dremio)} : des vues virtuelles Dremio exposant les résultats DQ historisés pour les outils de consommation.
    \item \textbf{Dashboard de suivi (Tableau)} : un tableau de bord interactif permettant aux Data Stewards et Data Owners de visualiser les KPIs de qualité, l'évolution temporelle des scores et la liste priorisée des anomalies à corriger.
\end{enumerate}

\subsection{Soda Core : bibliothèque Python de Data Quality}

\textbf{Soda Core} est une bibliothèque Python open source dédiée au contrôle de la qualité des données. Elle permet de définir des règles de qualité (appelées \textit{checks}) dans des fichiers de configuration YAML, de les exécuter sur différentes sources de données (Hive, Spark, PostgreSQL, Snowflake, etc.) et de collecter les résultats de manière programmatique. [REF]

Les avantages de Soda Core pour notre contexte sont les suivants :

\begin{itemize}
    \item \textbf{Compatibilité Hive / Spark} : Soda Core dispose d'un connecteur natif compatible avec l'environnement Hadoop de CIH Bank (\texttt{soda-core-spark}).
    \item \textbf{Configuration déclarative} : les règles DQ sont définies en YAML, ce qui facilite leur maintenance et leur évolution sans modifier le code Python.
    \item \textbf{Intégration Airflow} : Soda Core s'intègre nativement dans les DAGs Airflow via un opérateur Python, permettant de le déclencher automatiquement après chaque ingestion.
    \item \textbf{Résultats programmatiques} : les résultats des checks (valeur calculée, seuil, statut) sont accessibles en Python, ce qui permet de les persister facilement dans des tables Hive.
\end{itemize}

\subsection{Périmètre des contrôles : la zone RAW uniquement}

Le choix de cibler la \textbf{zone RAW} du Data Lake comme unique point de contrôle est délibéré et justifié par plusieurs raisons :

\begin{itemize}
    \item \textbf{Fidélité à la source} : la zone RAW (base Hive \texttt{raw\_nova\_referentieltiers}, chemin HDFS \texttt{/data/raw/novatiers/}) est la copie exacte et non transformée des tables \texttt{CCM\_*} issues d'Oracle NOVA. Contrôler la qualité sur RAW revient à mesurer directement la qualité du système opérant.
    \item \textbf{Isolation des responsabilités} : en contrôlant avant la transformation STAGING, on distingue clairement les anomalies issues du système source (responsabilité des Data Stewards Tiers) des éventuels artefacts introduits par les scripts de transformation.
    \item \textbf{Remédiation à la source} : les anomalies détectées sur RAW peuvent être remontées directement aux équipes saisie des agences pour correction dans NOVA, sans ambiguïté sur leur origine.
\end{itemize}

\subsection{Déclenchement automatique post-ingestion}

La solution s'insère dans le workflow Airflow existant. Une \textbf{tâche Python dédiée} (\texttt{run\_dq\_tiers\_checks}) est ajoutée dans le DAG \texttt{dailyow22}, en dépendance de toutes les tâches d'ingestion Sqoop des tables \texttt{CCM\_*}. Ainsi, les contrôles DQ se lancent \textbf{automatiquement chaque soir à 22h}, dès que les données RAW sont disponibles, sans intervention manuelle.

\subsection{Historisation et traçabilité quotidienne}

Les résultats de chaque exécution sont persistés dans deux tables Hive partitionnées par date (\texttt{run\_date}) dans une base dédiée \texttt{dq\_results} :

\begin{itemize}
    \item \textbf{\texttt{dq\_tiers\_scores}} : enregistre, pour chaque combinaison table/colonne/règle, la valeur calculée, le seuil configuré, le statut (OK/KO) et le score DQ normalisé sur 100.
    \item \textbf{\texttt{dq\_tiers\_anomalies}} : enregistre le détail des enregistrements clients (\texttt{identifier}, \texttt{radical}) en situation d'anomalie, avec la description de la règle violée et la criticité associée.
\end{itemize}

Cette historisation permet de mesurer l'\textbf{évolution temporelle} de la qualité des données à la source et de quantifier l'impact des corrections effectuées dans NOVA par les Data Stewards.

\section*{Conclusion}

Ce chapitre a présenté le contexte bancaire dans lequel s'inscrit ce projet, les enjeux critiques de la qualité des données KYC et CSP sur le Référentiel Tiers de CIH Bank, ainsi que les limites du pipeline de données actuel en matière de contrôle DQ. La solution proposée — un pipeline automatisé basé sur Soda Core, orchestré par Airflow, avec persistance dans Hive et restitution dans Tableau — répond directement à ces problématiques.
