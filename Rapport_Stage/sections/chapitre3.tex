\chapter{Conception de la Solution}

\section*{Introduction}
Ce chapitre présente la conception détaillée du pipeline de Data Quality sur le Référentiel Tiers de CIH Bank. Les contrôles de qualité s'appliquent \textbf{exclusivement à la zone RAW} du Data Lake, qui constitue la copie exacte et fidèle des tables \texttt{CCM\_*} issues du système opérant Oracle NOVA. Nous définirons l'architecture globale de la solution, le modèle de données pour la persistance des résultats DQ, le catalogue complet des règles de contrôle, ainsi que la conception des composants d'orchestration, d'exposition et de visualisation.

\section{Architecture globale de la solution}

\subsection{Vue d'ensemble}

La solution de Data Quality s'insère dans le pipeline existant du domaine Tiers comme une \textbf{couche transversale de contrôle} ciblant exclusivement la \textbf{zone RAW}. La logique est simple : la zone RAW est la copie conforme du système opérant Oracle NOVA. La solution comprend \textbf{cinq composantes principales} :

\begin{enumerate}
    \item \textbf{Source} : Oracle NOVA (Core Banking) --- inchangée
    \item \textbf{Ingestion RAW} : DAG Airflow \texttt{dailyow22} (22h) --- inchangée
    \item \textbf{Moteur DQ} : Script Python Soda Core (\texttt{dq\_tiers\_checks.py}) exécuté \textbf{sur la zone RAW} --- \textbf{nouveau}
    \item \textbf{Persistance} : Tables Hive de résultats DQ (base \texttt{dq\_results}) --- \textbf{nouveau}
    \item \textbf{Exposition \& Visualisation} : Vues Dremio \texttt{dq-tiers} + Dashboard Tableau --- \textbf{nouveau}
\end{enumerate}

\section{Catalogue des règles de Data Quality}

\subsection{Principes de conception des règles}

Les règles de Data Quality ont été définies selon les principes suivants :
\begin{itemize}
    \item \textbf{Fondement métier} : chaque règle est justifiée par un besoin métier concret (conformité réglementaire KYC, fiabilité du scoring CSP, détection d'anomalies de saisie).
    \item \textbf{Périmètre RAW} : toutes les règles s'appliquent sur les tables de la \textbf{zone RAW Hive} (base \texttt{raw\_nova\_referentieltiers}) qui sont la copie conforme des tables \texttt{CCM\_*} d'Oracle NOVA.
    \item \textbf{Seuils quantifiés} : chaque règle dispose d'un seuil d'alerte mesurable (ex : taux de valeurs nulles > 5\% = alerte).
    \item \textbf{Criticité graduée} : chaque règle est classée selon quatre niveaux de criticité : CRITIQUE, HAUTE, MOYENNE, FAIBLE.
    \item \textbf{Action corrective définie} : chaque règle est associée à une action corrective à destination des Data Stewards pour correction dans NOVA.
\end{itemize}

\subsection{Partitionnement et format de stockage}

Les deux tables sont \textbf{partitionnées par \texttt{run\_date}} (DATE), ce qui permet :
\begin{itemize}
    \item Une interrogation performante par période (dernier jour, 7 derniers jours, etc.)
    \item Une gestion du cycle de vie des données (rétention configurable)
    \item Un rechargement partiel en cas d'erreur sans affecter l'historique
\end{itemize}

Le format de stockage retenu est \textbf{ORC} (Optimized Row Columnar), cohérent avec les autres tables du Data Lake CIH Bank, pour une compression et des performances de requêtage optimales.

\section{Conception du moteur DQ — Soda Core}

\subsection{Architecture du script \texttt{dq\_tiers\_checks.py}}

Le script Python principal est organisé en trois phases :

\begin{enumerate}
    \item \textbf{Initialisation} : connexion à la source de données Hive via Soda Core, chargement des fichiers de configuration YAML des règles.
    \item \textbf{Exécution des checks} : Soda Core exécute les règles définies dans les fichiers YAML sur les tables Hive cibles et collecte les résultats.
    \item \textbf{Persistance} : les résultats sont formatés et écrits dans les tables \texttt{dq\_tiers\_scores} et \texttt{dq\_tiers\_anomalies} via une SparkSession.
\end{enumerate}

\subsection{Fichiers de configuration YAML}

Les règles DQ sont externalisées dans des fichiers YAML séparés par domaine, ce qui facilite leur maintenance sans modifier le code Python. Nous prévoyons trois fichiers :

\begin{itemize}
    \item \texttt{checks\_kyc.yml} : règles sur les colonnes KYC (\texttt{CCM\_NPDESCRIPTION})
    \item \texttt{checks\_csp.yml} : règles sur les colonnes CSP (\texttt{CCM\_JOB})
    \item \texttt{checks\_pipeline.yml} : règles de fraîcheur et cohérence pipeline
\end{itemize}

\section{Conception de l'orchestration Airflow}

\subsection{Intégration dans le DAG \texttt{dailyow22}}

La tâche DQ s'insère dans le DAG \texttt{dailyow22} existant, en \textbf{aval} de toutes les tâches d'ingestion des tables \texttt{CCM\_*}. Elle ne bloque pas le pipeline en cas d'erreur (mode \texttt{TriggerRule.ALL\_DONE}) afin de garantir que les résultats DQ sont produits même si certaines tables sont partiellement chargées.

\section{Conception du dashboard Tableau}

Le dashboard Tableau « \textbf{Qualité Données Référentiel Tiers} » est conçu pour deux profils d'utilisateurs :

\begin{itemize}
    \item \textbf{Data Owners et Direction} : vue exécutive avec scores globaux et alertes critiques
    \item \textbf{Data Stewards} : vue opérationnelle avec liste détaillée des anomalies à corriger
\end{itemize}

Le dashboard est structuré en \textbf{quatre onglets} :

\begin{enumerate}
    \item \textbf{Vue Exécutive} : Score DQ global KYC + CSP du jour, évolution sur 30 jours, alertes critiques en rouge.
    \item \textbf{Détail KYC} : Score par colonne KYC, heatmap par dimension DQ (Complétude / Validité / Cohérence), filtres par agence de rattachement.
    \item \textbf{Détail CSP} : Score CSP, analyse des valeurs parasites dans \texttt{employer}, taux de nullité par colonne, tendance d'amélioration après corrections.
    \item \textbf{Anomalies \& Remédiation} : Liste paginée des tiers en anomalie, triée par criticité, exportable pour les Data Stewards. Lien direct vers le \texttt{radical} CIH pour investigation dans NOVA.
\end{enumerate}
